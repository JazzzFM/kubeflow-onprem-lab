### Componentes empaquetados

```python
@dsl.component
def train_gradient_boosting(...) -> NamedTuple("GBCResult", ...): ...

@dsl.component
def train_cnn_pytorch(...) -> NamedTuple("CNNResult", ...): ...

@dsl.component
def compare_models(...) -> str: ...

@dsl.pipeline(name="ml-dl-demo-pipeline")
def ml_dl_pipeline(epochs: int = 2):
    gbc = train_gradient_boosting(...)
    cnn = train_cnn_pytorch(...)
    cnn.after(gbc)  # DAG dependency
    compare_models(
        gbc_acc=gbc.outputs["accuracy"],
        cnn_acc=cnn.outputs["accuracy"],
        ...
    )
```

### Setup runner

```python
from kfp import local
local.init(runner=local.SubprocessRunner(use_venv=False))
```

`use_venv=False` reutiliza el venv padre (donde ya están torch + sklearn),
evitando reinstalar 2 GB de torch por componente.

### Migración a EKS (próximo paso del curso)

El mismo código corre contra un cluster real con un solo cambio:

```python
# Antes (local):
from kfp import local
local.init(runner=local.SubprocessRunner())

# Después (EKS con KFP server):
from kfp import Client
client = Client(host="https://kfp.eks.example.com")
client.create_run_from_pipeline_func(ml_dl_pipeline, arguments={"epochs": 5})
```
